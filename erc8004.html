ERC-8004 on Quai

Agent identity, reputation, and validation registries for the Quai Network. Building the trust layer for AI-mediated commerce.

Overview

ERC-8004 defines three lightweight registries that enable agents to discover, authenticate, and trust each other across organizational boundaries. Quai's EVM compatibility means these contracts deploy directly, while Qi's energy-based value proposition adds a unique dimension: thermodynamic accountability.

◇
Identity Registry

ERC-721 with URIStorage. Each agent = NFT pointing to metadata and service endpoints.

◈
Reputation Registry

Standardized feedback signals. Scoring on-chain for composability, off-chain for richer algorithms.

◆
Validation Registry

Independent checks: stake-secured re-execution, zkML verifiers, TEE attestations.

Identity Registry

The Identity Registry uses ERC-721 with URIStorage extension. Each agent mints an NFT that serves as their on-chain identity. The token URI points to a registration file containing metadata and service endpoints.

Contract Interface

// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

import "@openzeppelin/contracts/token/ERC721/extensions/ERC721URIStorage.sol";

contract AgentIdentityRegistry is ERC721URIStorage {
    uint256 private _tokenIdCounter;
    
    // Agent registration fee (paid in Qi)
    uint256 public registrationFee = 10 ether; // 10 Qi
    
    event AgentRegistered(uint256 indexed tokenId, address indexed owner, string uri);
    event AgentUpdated(uint256 indexed tokenId, string newUri);
    
    constructor() ERC721("Quai Agent Identity", "QAGENT") {}
    
    function registerAgent(string memory _tokenURI) external payable returns (uint256) {
        require(msg.value >= registrationFee, "Insufficient registration fee");
        
        uint256 tokenId = _tokenIdCounter++;
        _safeMint(msg.sender, tokenId);
        _setTokenURI(tokenId, _tokenURI);
        
        emit AgentRegistered(tokenId, msg.sender, _tokenURI);
        return tokenId;
    }
    
    function updateAgentURI(uint256 tokenId, string memory _newURI) external {
        require(ownerOf(tokenId) == msg.sender, "Not agent owner");
        _setTokenURI(tokenId, _newURI);
        emit AgentUpdated(tokenId, _newURI);
    }
}

Reputation Registry

The Reputation Registry standardizes feedback signals. Each interaction can generate a reputation event that gets recorded on-chain. Aggregation can happen on-chain for composability or off-chain for more sophisticated algorithms.

Contract Interface

contract AgentReputationRegistry {
    struct FeedbackSignal {
        uint256 agentId;
        address submitter;
        int8 score;         // -100 to +100
        bytes32 taskHash;   // Hash of the task/interaction
        bytes32 paymentProof; // x402 transaction hash
        uint256 timestamp;
    }
    
    mapping(uint256 => FeedbackSignal[]) public agentFeedback;
    mapping(uint256 => int256) public aggregateScore;
    mapping(uint256 => uint256) public totalInteractions;
    
    event FeedbackSubmitted(
        uint256 indexed agentId,
        address indexed submitter,
        int8 score,
        bytes32 paymentProof
    );
    
    function submitFeedback(
        uint256 _agentId,
        int8 _score,
        bytes32 _taskHash,
        bytes32 _paymentProof
    ) external {
        require(_score >= -100 && _score <= 100, "Score out of range");
        
        agentFeedback[_agentId].push(FeedbackSignal({
            agentId: _agentId,
            submitter: msg.sender,
            score: _score,
            taskHash: _taskHash,
            paymentProof: _paymentProof,
            timestamp: block.timestamp
        }));
        
        // Simple running average (can be replaced with weighted algo)
        totalInteractions[_agentId]++;
        aggregateScore[_agentId] = 
            (aggregateScore[_agentId] * int256(totalInteractions[_agentId] - 1) + _score) 
            / int256(totalInteractions[_agentId]);
        
        emit FeedbackSubmitted(_agentId, msg.sender, _score, _paymentProof);
    }
}
Payment Proofs By requiring a paymentProof (x402 transaction hash), feedback is tied to actual economic interactions. This prevents sybil attacks and ensures reputation reflects real usage.

Validation Registry

The Validation Registry records independent verification checks. Developers can choose different trust models depending on risk tolerance:

Validation Type Mechanism Use Case
STAKE_SECURED Validator stakes Qi, re-executes task High-value transactions, financial data
ZKML Zero-knowledge proof of ML inference Model output verification without revealing inputs
TEE_ORACLE Trusted Execution Environment attestation Secure computation, sensitive data processing
CONSENSUS Multiple independent agents agree Decentralized fact-checking, data aggregation

Contract Interface

contract AgentValidationRegistry {
    enum ValidationType { STAKE_SECURED, ZKML, TEE_ORACLE, CONSENSUS }
    
    struct ValidationRecord {
        uint256 agentId;
        ValidationType valType;
        address validator;
        bytes32 taskHash;
        bytes proof;         // zkProof, TEE attestation, or stake tx
        bool passed;
        uint256 timestamp;
    }
    
    mapping(bytes32 => ValidationRecord[]) public taskValidations;
    mapping(address => uint256) public validatorStakes;
    
    uint256 public minStake = 1000 ether; // 1000 Qi minimum stake
    
    function stakeAsValidator() external payable {
        require(msg.value >= minStake, "Insufficient stake");
        validatorStakes[msg.sender] += msg.value;
    }
    
    function submitValidation(
        uint256 _agentId,
        ValidationType _valType,
        bytes32 _taskHash,
        bytes calldata _proof,
        bool _passed
    ) external {
        if (_valType == ValidationType.STAKE_SECURED) {
            require(validatorStakes[msg.sender] >= minStake, "Not staked");
        }
        
        taskValidations[_taskHash].push(ValidationRecord({
            agentId: _agentId,
            valType: _valType,
            validator: msg.sender,
            taskHash: _taskHash,
            proof: _proof,
            passed: _passed,
            timestamp: block.timestamp
        }));
    }
}

Quai-Specific Considerations

Energy-Based Accountability When agents pay each other in Qi, they're not just transferring tokens — they're transferring proof-of-work. This creates a natural Sybil resistance: spinning up fake agents to game reputation requires real energy expenditure.

Sharded Deployment

Quai's hierarchical structure (Prime → Region → Zone) creates deployment options:

Registry Recommended Chain Rationale
Identity Prime Chain Global namespace, cross-shard agent discovery
Reputation Zone Chains + Prime aggregator High-frequency writes on zones, periodic rollups to Prime
Validation Zone Chains Localized validation, lower latency
